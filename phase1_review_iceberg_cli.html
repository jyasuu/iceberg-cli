True Iceberg day-transform partitioning

The config and build_partition_spec in partition.rs already exist and are wired into table creation in engine.rs. The real gap is in overwrite: plan_overwrite does a full-table COW scan and filters by string-equality on the raw column value — it never uses the Iceberg PartitionSpec for scan pruning. The engine also drops + recreates the table on every overwrite, which defeats any partition layout already applied.

✓ build_partition_spec exists ✓ TableCreation wires spec ~ iceberg_partition config done ✗ scan pruning not used ✗ drop-recreate defeats spec blocker: COW scan is always full
write_strategies.rs:plan_overwrite engine.rs:COW snapshot block partition.rs:build_partition_spec
What to build: scope the overwrite scan to the incoming partition using Iceberg predicate pushdown; stop dropping + recreating the table; use fast_append to replace only that partition's files.

Position-delete files (MOR path)

The EqualityDeleteFileWriterBuilder and EqualityDeleteWriterConfig are already imported in write_strategies.rs. The module-level doc comment explains exactly why they can't be used yet: fast_append in iceberg-rust 0.9 validates DataContentType::Data only and rejects equality-delete files. The code structure for the future swap is already sketched out in the WritePlan struct (which carries a delete_files: Vec<DataFile> field).

✓ WritePlan.delete_files field ready ✓ EqualityDeleteWriter imported blocker: fast_append rejects delete content type ✗ no row_delta / add_delete_files in 0.9
iceberg-rust 0.9 Transaction API write_strategies.rs:WritePlan
What to build: keep COW for now; add a feature-flag code path that routes to MOR when add_delete_files lands upstream. The scaffolding (WritePlan.delete_files) is already in place.

Schema evolution: drop-column + rename

SchemaEvolutionConfig in config/mod.rs currently has a single field: allow_add_columns: bool. The engine in engine.rs:309 acts on this flag. The UpdateSchema transaction action needs to be called for drop and rename, but those fields don't exist on the config struct yet, and the engine has no branch for them.

✓ allow_add_columns implemented ✗ allow_drop_columns missing ✗ allow_rename_columns missing ✗ no engine branch for drop/rename
config/mod.rs:SchemaEvolutionConfig engine.rs:309 schema evolution
What to build: add allow_drop_columns and allow_rename_columns: HashMap<String, String> to SchemaEvolutionConfig; add corresponding UpdateSchema actions in the engine's schema-evolution block.
